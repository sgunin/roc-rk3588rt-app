◆ MV_CC_EnumDevices()

MV_CAMCTRL_API int __stdcall MV_CC_EnumDevices ( IN unsigned int  nTLayerType,
IN OUT MV_CC_DEVICE_INFO_LIST *  pstDevList 
)

Enumerate devices, including cameras on the corresponding frame grabber.

Parameters
nTLayerType[IN] Transport layer protocol type. See details of parameter definitions in CameraParams.h
Transport Layer Protocol Type Definition Value Description
MV_GIGE_DEVICE 0x00000001 GigE Vision frame grabber
MV_CAMERALINK_INTERFACE 0x00000004 Camera Link frame grabber
MV_CXP_INTERFACE 0x00000008 CoaX Press frame grabber
MV_XOF_INTERFACE 0x00000010 XoF Link frame grabber
pstDevList[IN][OUT] Device List
Returns
Return MV_OK for success, and return corresponding error code for failure.
Remarks
The memory of storing device list is allocated by the SDK and will be released and applied for during multiple-threaded API calling. It is recommended to avoid the multiple-threaded enumeration.
It is recommended to avoid multithreaded enumeration operations as much as possible. The transport layers of enumeration supports importing parameters including MV_GIGE_DEVICE, MV_1394_DEVICE, MV_USB_DEVICE, MV_CAMERALINK_DEVICE, and MV_GIGE_DEVICE sends out information of all GigE devices (including virtual GigE and GigE devices on GenTl),; The transport layers of MV_GENTL_GIGE_DEVICE, MV_GENTL_CAMERALINK_DEVICE, MV_GENTL_CXP_DEVICE, and MV_GENTL_XOF_DEVICE can return the camera information under the corresponding frame grabber.
Examples
ChunkData.cpp, Display.cpp, Events.cpp, ForceIP.cpp, ForceIPEx.cpp, Grab_Asynchronous.cpp, GrabImage.cpp, GrabImage_Callback.cpp, GrabImage_HighPerformance.cpp, GrabMultipleCamera.cpp, HighBandwidthDecode.cpp, ImageProcess.cpp, ImageSave.cpp, MultiCast.cpp, MultiLightCtrl_ImageStitching.cpp, ParametrizeCamera_AreaScanIOSettings.cpp, ParametrizeCamera_FileAccess.cpp, ParametrizeCamera_LineScanIOSettings.cpp, ParametrizeCamera_LoadAndSave.cpp, QuickSoftwareTrigger.cpp, ReconnectDemo.cpp, SetParam.cpp, Trigger_Image.cpp, and Trigger_ImageCallback.cpp.

MV_GENTL_XOF_DEVICE 0x00000200 XoF device.

The transport layers of MV_GENTL_GIGE_DEVICE, MV_GENTL_CAMERALINK_DEVICE, MV_GENTL_CXP_DEVICE, and MV_GENTL_XOF_DEVICE can return the camera information under the corresponding frame grabber.

◆ MV_CC_EnumDevicesEx2()

MV_CAMCTRL_API int __stdcall MV_CC_EnumDevicesEx2 ( IN unsigned int  nTLayerType,
IN OUT MV_CC_DEVICE_INFO_LIST *  pstDevList,
IN const char *  strManufacturerName,
IN MV_SORT_METHOD  enSortMethod 
)

Enumerate devices on the corresponding frame grabber according to the specified sorting type and the manufacturer name.

Parameters
nTLayerType[IN] Transport layer protocol type. Different types of devices are distinguished. See details of parameter definitions in CameraParams.h
Transport Layer Protocol Type Definition Value Description
MV_GIGE_DEVICE 0x00000001 GigE device.
MV_USB_DEVICE 0x00000004 USB device.
MV_CAMERALINK_DEVICE 0x00000008 CameraLink device (serial port).
MV_VIR_GIGE_DEVICE 0x00000010 Virtual GigE device.
MV_VIR_USB_DEVICE 0x00000020 Virtual USB device.
MV_GENTL_GIGE_DEVICE 0x00000040 Self-developed NIC GigE device.
MV_GENTL_CAMERALINK_DEVICE 0x00000080 CameraLink device.
MV_GENTL_CXP_DEVICE 0x00000100 CoaXPress device.
MV_GENTL_XOF_DEVICE 0x00000200 XoF device.
pstDevList[IN][OUT] Device list.
strManufacturerName[IN] Manufacturer name. It can be set to NULL, which indicates to enumerate all devices according to the specified sorting type.
enSortMethod[IN] Sorting type.
Returns
Return MV_OK for success, and return corresponding error code for failure..
Remarks
The memory of storing device list is allocated by the SDK and will be released and applied for during multiple-threaded API calling. It is recommended to avoid the multiple-threaded enumeration.
strManufacturerName can be set to NULL, which indicates to enumerate all devices according to the specified sorting type; if not set to NULL, only the sorted specified manufacturer device list will be returned.
The transport layers of MV_GENTL_GIGE_DEVICE, MV_GENTL_CAMERALINK_DEVICE, MV_GENTL_CXP_DEVICE, and MV_GENTL_XOF_DEVICE can return the camera information under the corresponding frame grabber.
The differences between MV_CC_EnumDevicesEx2() and MV_CC_EnumDevicesEx() are as follows:
  • If MV_GIGE_DEVICE is configured for MV_CC_EnumDevicesEx(), virtual network interface cameras and network interface cameras under self-developed frame grabber will be enumerated besides the network interface cameras in the network segment; if MV_USB_DEVICE is configured, USB cameras and virtual USB cameras will be enumerated.
  • If MV_GIGE_DEVICE is configured for MV_CC_EnumDevicesEx2(), only network interface cameras in the network segment will be enumerated; if MV_USB_DEVICE is configured, only USB cameras will be enumerated.
  • MV_CC_EnumDevicesEx2() contains an extra sorting funtion. The sorting type is determined by the parameter MV_SORT_METHOD enSortMethod.

◆ MV_CC_IsDeviceAccessible()

MV_CAMCTRL_API bool __stdcall MV_CC_IsDeviceAccessible ( IN MV_CC_DEVICE_INFO *  pstDevInfo,
IN unsigned int  nAccessMode 
)

Check if the specified device can be accessed.

Parameters
pstDevInfo[IN] Device information.
nAccessMode[IN] Access type.
Macro Definition Value Description
MV_ACCESS_Exclusive 1 Exclusive permission, for other apps, the CCP register is only allowed to be read
MV_ACCESS_ExclusiveWithSwitch 2 Preempt permission in mode 5, and then open with exclusive permission
MV_ACCESS_Control 3 Control permission, for other apps, all registers are allowed to be read
MV_ACCESS_ControlWithSwitch 4 Preempt permission in mode 5, and then open with control permission
MV_ACCESS_ControlSwitchEnable 5 Open with control permission that can be preempted
MV_ACCESS_ControlSwitchEnableWithKey 6 Preempt permission in mode 5, and then open with control permission that can be preempted
MV_ACCESS_Monitor 7 Open device with reading mode, suitable under control permission
Returns
Return true to indicate the device is accessible, and return false to indicate no permission or the device is offline.
Remarks
You can read the device CCP register value to check the current access permission.
Return false if the device does not support the modes MV_ACCESS_ExclusiveWithSwitch, MV_ACCESS_ControlWithSwitch, MV_ACCESS_ControlSwitchEnableWithKey. Currently the device does not support the 3 preemption modes, neither do the devices from other mainstream manufacturers.
This API is not supported by CameraLink device.
Examples
ForceIP.cpp, and ForceIPEx.cpp.

◆ MV_CC_OpenDevice()

MV_CAMCTRL_API int __stdcall MV_CC_OpenDevice ( IN void *  handle,
IN unsigned int  nAccessMode,
IN unsigned short  nSwitchoverKey 
)

Open a device.

Parameters
handle[IN] Device handle
nAccessMode[IN] Access Right
Macro Definition Value Description
MV_ACCESS_Exclusive 1 Exclusive permission, for other apps, the CCP register is only allowed to be read
MV_ACCESS_ExclusiveWithSwitch 2 Preempt permission in mode 5, and then open with exclusive permission
MV_ACCESS_Control 3 Control permission, for other apps, all registers are allowed to be read
MV_ACCESS_ControlWithSwitch 4 Preempt permission in mode 5, and then open with control permission
MV_ACCESS_ControlSwitchEnable 5 Open with control permission that can be preempted
MV_ACCESS_ControlSwitchEnableWithKey 6 Preempt permission in mode 5, and then open with control permission that can be preempted
MV_ACCESS_Monitor 7 Open device with reading mode, suitable under control permission
nSwitchoverKey[IN] Key for switching permissions.
Returns
Return MV_OK for success, and return corresponding error code for failure..
Remarks
You can find the specific device and connect according to inputted device parameters.
When calling the interface, the parameters nAccessMode and nSwitchoverKey are optional, and the device access mode is exclusive by default. Currently the device does not support the following preemption modes: MV_ACCESS_ExclusiveWithSwitch, MV_ACCESS_ControlWithSwitch, MV_ACCESS_ControlSwitchEnableWithKey.
For USB3Vision device, the parameters nAccessMode and nSwitchoverKey are invalid.
Examples

◆ MV_CC_GetAllMatchInfo()

MV_CAMCTRL_API int __stdcall MV_CC_GetAllMatchInfo ( IN void *  handle,
IN OUT MV_ALL_MATCH_INFO *  pstInfo 
)

Get the information of all types.

Parameters
handle[IN] Device handle.
pstInfo[IN][OUT] Structure pointer of various type of information.
Returns
Return MV_OK for success, and return corresponding error code for failure..
Remarks

  • Input required information type (specify nType in structure MV_ALL_MATCH_INFO) in the API and get corresponding information (return in pInfo of structure MV_ALL_MATCH_INFO).

  • The calling precondition of this API is determined by obtained information type. Call after enabling capture to get :MV_MATCH_TYPE_NET_DETECT information of GigE device, and call after starting device to get MV_MATCH_TYPE_USB_DETECT information of USB3Vision device.
  • This API is not supported by CameraLink device.
